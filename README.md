# 🧠 분석 스토리｜떡락하는 채용 플랫폼.. 살리고 싶어..

## 📌 상황 
> 대형 채용 플랫폼에 밀려 **DAU 하락, 공고 수 하락**...  
> "우리 플랫폼, 왜 이렇게 사람들이 안 눌러?"

- 사용자도, 기업도 대형 플랫폼으로 이동 중  
- 자사 플랫폼의 **전환률이 심각하게 낮아짐**
- UI/UX 문제? 콘텐츠 문제?  
→ **원인을 행동 데이터로 파악하는 것부터 시작**

--- 

### ✅ 과제 

> **“공고 방문은 했는데, 왜 지원은 안 했지?”**  
> 문제는 ‘지원하기’ 버튼 앞에서 멈춘 사용자였다!

그래서 제가 집중한 핵심 과제는 다음과 같았습니다

- 행동 로그를 바탕으로 **사용자 이탈 퍼널을 정의하고 분석**
- 이탈 원인을 추정하고, **UI/UX 개선 가설을 설계**
- 클릭률 향상을 검증하기 위한 **A/B 테스트 기획 및 조건 설정**
- 다양한 사용자 유형을 시각화하기 위한 **RFM 기반 대시보드 구현**

--- 

### ✅ 실행

#### 🔹 1. 퍼널 정의 & 로그 해부  
> **"지원까지 몇 번의 클릭이 필요한가?"**

- 사용자 행동 흐름을 파악하기 위해 **채용 여정을 퍼널로 재구성**했습니다.  
- 로그 데이터에서 **URL path와 query string을 직접 파싱**하여  
  `채용페이지 → 공고페이지 → 지원시작 → 지원완료`의 전환 흐름을 정의했습니다.  
- 분석 결과, 가장 큰 이탈은 **공고페이지 → 지원시작 구간에서 무려 37%** 발생!  
- 사용자가 공고를 ‘쇼핑하듯’ 둘러보다 떠나는 패턴을 발견했고,  
  **UX 개선이 필요하다는 인사이트**를 도출했습니다.

---

#### 🔹 2. A/B 테스트 설계  
> **"버튼 위치 하나가 전환율을 바꾼다?"**

- 이탈 원인을 검증하기 위해 **구체적인 행동 기반 가설 2가지를 설정**했습니다:

  - **가설 1**: ‘지원하기’ 버튼이 **스크롤 시 사라져** 클릭률이 낮다  
    → ✅ **버튼 고정 UI 실험**

  - **가설 2**: **직군 필터가 부실**해 사용자가 원하는 공고를 찾지 못한다  
    → ✅ **개발직군 필터 개선 실험**

- 각 실험에 대해:
  - **성공 지표 / 보조 지표 / 가드레일 지표**를 정의하고
  - **DAU 기반 샘플 수 산정**
  - **노출 기간을 설계**해 실험 기반을 완성했습니다.

---

#### 🔹 3. 사용자 세분화 및 시간 흐름 분석  
> **"누가 계속 오고, 누가 다시 안 올까?"**

- 단순 이탈률만으로는 부족했기에, **사용자를 유형별로 나누고 흐름을 분석**했습니다.

  - **RFM 모델**을 적용해 유저를  
    → **신규 / 휴면 / 충성 고객**으로 세분화

  - 각 유형에 대해  
    → **시간대별 지원 패턴**  
    → **공고 수 대비 지원률의 관계**를 분석

- 그 결과, “**언제 어떤 사용자가 어떤 행동을 보이는지**”에 대한 구체적 인사이트를 확보했습니다.

---

#### 🔹 4. 대시보드 설계 & 시각화  
> **"한눈에 보는 이탈과 전환의 지도"**

- 비즈니스 팀이 실시간으로 활용할 수 있도록 **직관적인 대시보드를 직접 설계**했습니다.
  - UI는 **Figma로 와이어프레임 제작**
  - 구현은 **Tableau로 인터랙티브 대시보드 구성**

- 제작한 대시보드:
  - 📊 **퍼널 대시보드**  
    → 주요 이탈 구간 시각화 및 필터링 기능
  - 👤 **RFM 대시보드**  
    → 사용자 유형별 전환률 & 행동 흐름 확인

- **실시간 필터 적용**이 가능하며,  
  **데이터 기반 전략 수립을 위한 실무 활용이 용이**하도록 설계했습니다.
--- 

### ✅ 결과  
> **숫자, 실험, 시각화 — 전환을 향한 3단 로드맵 완성**

| 🏆 주요 성과 항목         | 🚀 성과 내용                                                                 |
| ------------------------- | --------------------------------------------------------------------------- |
| 퍼널 정의 및 이탈 분석     | 사용자 여정 중 **가장 큰 이탈 구간(공고 → 지원 시작, 37%)**을 도출하고 <br> 이탈 원인을 **탐색 행동 패턴**으로 분석 |
| A/B 테스트 실험 설계      | **‘지원하기’ 버튼 고정**, **직군 필터 개선** 등 <br> 실제 개선을 위한 **가설 기반 A/B 테스트** 설계 완료 |
| 데이터 시각화 및 대시보드 | **Tableau 대시보드**로 실시간 사용자 흐름과 세그먼트 분석 가능 <br> → **의사결정 속도와 명확성 향상** |

## 💡 회고 

### ✅ 잘한 점｜**가설에서 공감까지, A/B 테스트 설계로 이끌어낸 전략적 사고**

- 단순 분석을 넘어, **사용자 행동을 구체적인 문제 상황으로 연결**한 점이 높은 평가를 받았습니다.
- 특히 ‘**스크롤 시 사라지는 지원 버튼**’과 ‘**직무 필터 부족**’이라는 가설은  
  사용자 입장에서 "진짜 이거 고치면 클릭률 오를 것 같다!"는 공감을 이끌어냈습니다.
- 실험 설계 시에도 단순히 가설만 세운 것이 아니라,
  - **기존안 vs 개선안 정리**
  - **성공/보조/가드레일 지표 명확화**
  - **DAU 기반 샘플 사이즈 산정**까지
  → 실제 서비스에서 바로 적용 가능한 수준으로 설계했습니다.

---

### ❗ 아쉬웠던 점｜**한 발 더 들어갔으면 보였을 ‘탐색 시간’의 힌트들**

- 탐색 과정떄문에 이탈률이 높다는 사실은 파악했지만,  
  그 **이탈을 유발한 ‘탐색 과정’ 자체에 더 깊이 들어가보지 못한 점**은 아쉬웠습니다.
- 예를 들어, 평균 탐색 시간이나 비교 클릭 수를 분석했다면  
  “불필요한 탐색 과정을 어떻게 줄일 것인가”라는 문제정의로 확장할 수 있었을 것입니다.
- 또한 여러 개의 A/B 테스트 가설을 설계했지만,  
  실제 서비스 상황이라면 **하나의 테스트에 집중하여 명확하게 검증하는 전략**이 더 효과적이라는 점을 느꼈습니다.

---

### 🔄 다음엔 이렇게!｜**실험 이전의 아이디어, 실험 이후의 전략까지**

- A/B 테스트가 모든 문제의 정답은 아닙니다.  
  앞으로는 "전환을 높이기 위한 다양한 해결 방식"을 먼저 폭넓게 아이디에이션하고,  
  그중에서 리소스와 임팩트를 고려해 가장 적합한 방법을 선택하는 사고가 필요하다고 느꼈습니다.

- 단순히 "이탈률을 줄이자"가 아니라,  
  "이탈을 줄인 뒤 어떤 데이터를 활용해 어떤 전략을 펼칠 수 있을까?"까지  
  고민하는 **데이터드리븐 사고 전환**이 중요하다고 느꼈습니다.
